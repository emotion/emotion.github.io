## 背景

最近遇到一个 MySQL 主从不同步的问题。

线上有两个模块 A 和 B。模块 A 有三台机器，并在本机部署了 MySQL，整体是一主二从的结构。

模块 B 需要读取模块 A 的数据。这个读取链路有几个特点：

1. 可以接受一定延迟；
2. 单次读取的数据量可能比较大；
3. 如果通过模块 A 的接口读取，容易导致接口超时；
4. 当时框架不支持单独调大某个接口的超时时间。

所以当时的方案是：**模块 B 直接读取模块 A 的从库数据**。

这个设计本身还能接受，因为它只是读从库。真正的问题出现在后面。

随着业务发展，模块 B 新增了写入数据的接口。当时没有仔细梳理整体架构，也没有留下清晰文档，于是就在模块 A 的从库上新建了一张 MySQL 表，并把模块 B 的数据写入了这张从库上的表。

后来我要变更数据库表结构时，才发现这张表只存在于从库，不存在于主库。

这就比较尴尬了。

因为 MySQL 主从同步的正确方向应该是：

```text
主库 -> 从库
```

而现在实际变成了：

```text
主库 -> 从库
       从库上还有额外业务写入
```

这样会导致几个问题：

1. 主库没有这张表，主从结构已经不一致；
2. 从库上存在业务写入，后续主从同步可能因为同名表、同名数据或 DDL 冲突出问题；
3. 如果以后要重新搭建从库，或者从主库恢复数据，这张表的数据会丢失；
4. 架构语义混乱：从库本来应该是只读副本，却承载了写入职责。

为了不继续留下这个坑，最终决定把这张表和相关数据迁回主库，让主从数据库重新恢复一致。

---

## 当时比较幸运的条件

这次能用比较直接的方式修复，是因为业务条件还算宽松：

1. 模块 B 可以暂停一段时间不吐数据，只要后续恢复即可；
2. 只有模块 C 会从模块 B 拉取这部分数据；
3. 这张表的数据只有插入和拉取，没有更新；
4. 数据写入量比较少；
5. 即使缺失少量数据，业务影响也相对可控。

这些条件很关键。

如果这张表存在大量更新、强一致依赖，或者有多个下游实时消费方，那处理方案就不能这么简单，需要更严格的数据校验、双写切换、回滚预案，甚至可能要做临时迁移程序。

---

## 处理目标

这次修复的目标是：

1. 把只存在于从库的表结构同步到主库；
2. 把从库中已经写入的历史数据迁移到主库；
3. 将模块 B 的写入入口从从库切换到主库；
4. 删除从库上的脏表，让它重新通过主从同步从主库获得这张表和数据；
5. 验证主从数据一致；
6. 恢复模块 C 的数据拉取。

最终希望恢复成正常结构：

```text
模块 B 写入 -> 主库 -> 从库
模块 C 读取 -> 模块 B / 从库
```

---

## 操作前检查

正式操作前，先确认当前主从状态。

在从库上查看同步状态：

```sql
SHOW SLAVE STATUS\G
```

重点关注：

```text
Slave_IO_Running: Yes
Slave_SQL_Running: Yes
Seconds_Behind_Master: xxx
Master_Log_File: mysql-bin.xxxxxx
Read_Master_Log_Pos: yyyy
Relay_Master_Log_File: mysql-bin.xxxxxx
Exec_Master_Log_Pos: zzzz
Last_Error: ...
```

如果是 MySQL 8.0，也可能使用：

```sql
SHOW REPLICA STATUS\G
```

还需要确认：

1. 主库是否确实没有这张表；
2. 从库这张表的数据量；
3. 从库这张表是否有主键或唯一索引；
4. 表是否只有 insert，没有 update/delete；
5. 模块 B 当前写入的是哪台机器；
6. 模块 C 当前从哪里拉取数据；
7. 是否有其它隐藏消费者。

---

## 整体方案

最终处理步骤如下：

1. 先上线模块 C，暂停从模块 B 拉取数据；
2. 停止模块 B 对从库这张表的写入，或者让模块 B 暂停吐数据；
3. 记录从库当前同步位点；
4. 暂停从库主从同步；
5. 从从库 dump 表结构和历史数据；
6. 将 dump 数据导入主库；
7. 将模块 B 的写入切换到主库；
8. 导出暂停期间从库上新增的数据，补写到主库；
9. 删除从库上的这张表；
10. 重新开启从库同步，让主库上的表结构和数据同步回从库；
11. 校验主从数据一致；
12. 恢复模块 C 的拉取。

这里最重要的是两点：

1. **切写入方向**：以后只能写主库，不能再写从库；
2. **补齐切换窗口数据**：dump 到切写入之间产生的数据不能漏。

---

## 具体操作步骤

下面用 `target_table` 表示这张误写到从库的表。

### 1. 暂停下游拉取

先暂停模块 C 从模块 B 拉取数据。

原因是迁移过程中，模块 B 的数据来源和写入方向会发生变化，如果模块 C 在中间状态读取，可能读到不完整或重复的数据。

这个动作最好通过配置开关完成，而不是直接停服务。

---

### 2. 暂停模块 B 写入

如果业务允许，先暂停模块 B 对这张表的写入。

这是最简单、风险最低的方式。

如果不能暂停写入，就需要做双写或增量同步，复杂度会高很多。

---

### 3. 记录从库同步位点

在从库执行：

```sql
SHOW SLAVE STATUS\G
```

记录下面几个字段：

```text
Relay_Master_Log_File
Exec_Master_Log_Pos
```

这两个字段表示 SQL 线程已经执行到主库 binlog 的哪个位置。

也可以同时记录：

```text
Master_Log_File
Read_Master_Log_Pos
```

这样后续如果需要恢复或排查，可以知道当时主从同步停在什么位置。

---

### 4. 暂停从库同步

在从库执行：

```sql
STOP SLAVE;
```

MySQL 8.0 可以使用：

```sql
STOP REPLICA;
```

暂停后再次确认：

```sql
SHOW SLAVE STATUS\G
```

确认 SQL 线程已经停止。

---

### 5. 从从库 dump 表结构和数据

从从库导出这张表：

```bash
mysqldump -h从库地址 -u用户 -p \
  --single-transaction \
  --set-gtid-purged=OFF \
  数据库名 target_table > target_table.sql
```

如果表很小，也可以拆成结构和数据两部分：

```bash
mysqldump -h从库地址 -u用户 -p \
  --no-data \
  --set-gtid-purged=OFF \
  数据库名 target_table > target_table_schema.sql

mysqldump -h从库地址 -u用户 -p \
  --no-create-info \
  --single-transaction \
  --set-gtid-purged=OFF \
  数据库名 target_table > target_table_data.sql
```

如果表只有 insert，没有 update/delete，而且数据量不大，这种方式比较直接。

---

### 6. 导入主库

先在主库确认表不存在：

```sql
SHOW TABLES LIKE 'target_table';
```

然后导入：

```bash
mysql -h主库地址 -u用户 -p 数据库名 < target_table.sql
```

导入后检查数量：

```sql
SELECT COUNT(*) FROM target_table;
```

如果有主键，也可以检查最大 ID 或时间范围：

```sql
SELECT MIN(id), MAX(id), COUNT(*) FROM target_table;
```

---

### 7. 将模块 B 写入切到主库

修改模块 B 的数据库写入配置，让它写主库。

切换后要验证：

1. 新写入的数据出现在主库；
2. 从库没有继续被模块 B 直接写入；
3. 主库写入后，数据能通过主从同步同步到从库。

这一步是整个修复的核心。

如果这一步没做好，即使本次数据修好了，后面还会继续产生新的主从不一致。

---

### 8. 补齐切换窗口数据

如果在 dump 完成之后，到模块 B 写入切主库之前，从库上还有新增数据，需要把这段窗口数据补到主库。

因为这张表只有 insert，所以可以按自增 ID 或创建时间补数据。

比如 dump 时记录了最大 ID：

```sql
SELECT MAX(id) FROM target_table;
```

假设当时最大 ID 是 `10000`，切换前从库又写入了一些新数据，那么可以导出：

```sql
SELECT * FROM target_table WHERE id > 10000;
```

或者用 `mysqldump --where`：

```bash
mysqldump -h从库地址 -u用户 -p \
  --no-create-info \
  --set-gtid-purged=OFF \
  --where="id > 10000" \
  数据库名 target_table > target_table_increment.sql
```

然后导入主库：

```bash
mysql -h主库地址 -u用户 -p 数据库名 < target_table_increment.sql
```

如果没有可靠的自增 ID，就需要用创建时间、业务唯一键，或者导出后用脚本做去重合并。

---

### 9. 删除从库上的脏表

确认主库已经有完整数据，并且模块 B 已经切到主库写入后，再处理从库。

从库上这张表是错误来源，需要删除，让它后续从主库同步回来。

```sql
DROP TABLE target_table;
```

注意，这一步一定要在确认主库已经有表和数据之后再做。

如果不放心，可以先把从库上的表 rename 备份：

```sql
RENAME TABLE target_table TO target_table_backup_20210501;
```

等主从同步和数据校验都完成后，再删除备份表。

这种方式比直接 drop 更稳。

---

### 10. 重新开启主从同步

在从库执行：

```sql
START SLAVE;
```

MySQL 8.0 可以使用：

```sql
START REPLICA;
```

然后查看状态：

```sql
SHOW SLAVE STATUS\G
```

确认：

```text
Slave_IO_Running: Yes
Slave_SQL_Running: Yes
Last_Error: 为空
Seconds_Behind_Master: 逐渐追平
```

如果前面主库已经创建了 `target_table`，从库删除旧表后，主库上的 DDL 和数据应该会重新同步到从库。

---

### 11. 数据校验

至少做以下几类校验：

```sql
SELECT COUNT(*) FROM target_table;
SELECT MIN(id), MAX(id) FROM target_table;
SELECT MIN(create_time), MAX(create_time) FROM target_table;
```

分别在主库和从库执行，对比结果。

如果有业务唯一键，也可以抽样校验：

```sql
SELECT * FROM target_table WHERE id IN (...);
```

更严格一点，可以按主键范围分段统计：

```sql
SELECT FLOOR(id / 10000) AS bucket, COUNT(*)
FROM target_table
GROUP BY FLOOR(id / 10000)
ORDER BY bucket;
```

如果每个分段的数量都一致，基本可以确认没有明显缺失。

---

### 12. 恢复模块 C 拉取

确认主库、从库数据一致后，恢复模块 C 的拉取。

恢复后继续观察：

1. 模块 C 是否能正常拉到数据；
2. 模块 B 是否只写主库；
3. 从库同步是否正常；
4. 是否有重复数据或缺失数据告警。

---

## 风险点

这类问题最容易出错的地方有几个。

### 1. 从库上还有写入

如果模块 B 没有完全停止从库写入，就可能在迁移过程中继续产生新数据。

所以一定要确认写入入口已经切走。

### 2. dump 和切换之间的数据遗漏

如果 dump 时没有停写，或者停写不彻底，中间窗口的数据要单独补齐。

这也是为什么最好在 dump 前暂停写入。

### 3. 主从同步遇到 DDL 冲突

如果从库上已经存在同名表，主库同步过来的 `CREATE TABLE` 可能失败。

所以重新同步前，需要先删除或 rename 从库上的旧表。

### 4. 直接跳过错误会埋坑

遇到主从同步错误时，有些场景可以通过跳过一个事务恢复同步。

但这次问题不是单条 SQL 执行失败，而是主从结构已经不一致。

如果只是简单跳过错误，数据结构仍然不一致，后续还会继续出问题。

### 5. 从库不应该承载业务写入

这才是根因。

如果确实需要模块 B 写入数据，就应该写主库，或者给模块 B 自己的库表，而不是写到模块 A 的从库上。

---

## 复盘

这次问题的根因不是 MySQL 主从同步机制本身，而是系统边界没有定义清楚：

1. 模块 B 为了规避接口超时，直接读取了模块 A 的从库；
2. 后续模块 B 又在这个从库上新增表并写入数据；
3. 从库从只读副本变成了业务写入点；
4. 但是主从同步方向仍然是主库到从库，导致主库缺表、主从结构不一致。

这里有几个教训：

1. 从库原则上只读，不应该承载业务写入；
2. 跨模块直接读数据库要谨慎，最好有清晰边界和文档；
3. 如果因为性能或超时绕过接口，要补上架构说明；
4. 数据库表结构变更前，应先确认主从拓扑和真实读写入口；
5. 文档真的很重要，否则后面接手的人很容易踩坑。

---

## 小结

这次处理的核心不是简单地“修主从同步”，而是把错误的写入方向纠正回来。

最终方案可以概括为：

```text
暂停下游读取
-> 暂停从库写入
-> dump 从库脏表数据
-> 导入主库
-> 模块 B 写入切主库
-> 补齐切换窗口数据
-> 删除或 rename 从库旧表
-> 重新开启主从同步
-> 校验主从一致
-> 恢复下游读取
```

如果业务允许暂停写入，这个方案比较简单可靠。

如果业务不允许停写，就不能直接这么做，需要设计双写、增量同步、幂等导入和回滚方案，复杂度会高很多。
